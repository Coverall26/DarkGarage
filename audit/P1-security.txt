SECTION 4: DEFENSE-IN-DEPTH ARCHITECTURE
================================================================================

The platform implements 5 layers of auth enforcement:

  Layer 1: proxy.ts edge middleware
           - JWT validation for ALL /api/ routes via edge-auth.ts
           - Route classification (PUBLIC/CRON/AUTHENTICATED/TEAM_SCOPED/ADMIN)
           - Blanket rate limiting (200 req/min/IP)

  Layer 2: proxy.ts admin auth
           - Additional JWT + LP blocking for /admin/* and /api/admin/*
           - User context headers injected (x-middleware-user-id/email/role)

  Layer 3: Route handler RBAC
           - requireAdminAppRouter() on 40+ admin route files (43 handlers)
           - requireLPAuthAppRouter() on sensitive LP routes
           - enforceRBAC() / requireAdmin() on Pages Router routes
           - authenticateGP() for marketplace/fund routes

  Layer 4: DomainMiddleware
           - Domain-level gating for app.admin.fundroom.ai

  Layer 5: Business logic auth
           - Team-scoped Prisma queries (teamId/orgId in WHERE clauses)
           - Fund ownership verification
           - Investor profile ownership checks

MEDIUM: 2

  M-1. new Function() for formula evaluation
       File: lib/esign/field-types.ts:455
       Code: new Function(`"use strict"; return (${expression})`)()
       Context: Used for e-signature FORMULA field type evaluation
       Mitigation: Input is first:
         1. Field references replaced with numeric values
         2. Whitelist regex: /^[\d\s+\-*/().]+$/ (numbers, ops, parens only)
         3. Wrapped in try-catch, returns "ERROR" on any exception
         4. Result validated: must be finite number
       Risk: The regex whitelist is effective against arbitrary code execution.
         However, Function() constructor is generally considered unsafe and could
         be bypassed if the regex is later relaxed. A dedicated math evaluator
         library (e.g., mathjs) would be safer.
       Recommendation: Replace new Function() with a dedicated safe math evaluator
         in a future sprint. Current mitigation is acceptable for MVP launch.

  M-2. Offering page custom CSS injection surface
       File: app/offering/[slug]/page-client.tsx:288
       Code: <style dangerouslySetInnerHTML={{ __html: sanitizeCss(data.customCss) }} />
       Context: GP-configured custom CSS for offering landing pages
       Mitigation: sanitizeCss() strips url(), @import, expression(), javascript:,
         -moz-binding, behavior:, and existing comments. Returns sanitized string.
       Risk: CSS can still be used for data exfiltration in some browsers via
         background-image on input[value^="X"] selectors, though this is a
         limited attack surface. The customCss field is GP-controlled (admin only),
         not user-controlled.
       Recommendation: Consider using a CSS-in-JS approach or CSS modules instead
         of raw CSS injection. Acceptable for MVP since only GP admins can set
         custom CSS.

LOW: 1

  L-1. Duplicate health endpoints
       Files: pages/api/health.ts + app/api/health/route.ts
       Issue: Two health check endpoints exist (Pages Router + App Router).
         Both are functionally identical. The Pages Router version should be
         removed during the Phase 2 router migration.
       Risk: Negligible — both return the same non-sensitive service status.
       Recommendation: Remove pages/api/health.ts after verifying App Router
         version is the canonical monitoring target.
